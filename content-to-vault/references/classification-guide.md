# 分类指南

仅当确定 category、tags、topics、wiki 字段时读取本文件。在生成正文和来源与素材段之后再加载；正文撰写阶段不要读取本文件。

## category

路径式单字段，斜杠分隔层级，如 `技术/编程语言`、`生活/数码设备`、`商业/公司分析`。表达笔记的主检索路径，与 Obsidian 目录结构一致。

> 目录路径表达主检索路径。topics 和 tags 表达交叉属性。

### 分类判断逻辑

分类必须自动完成，不向用户确认。

分类不是固定目录匹配，而是根据笔记主要价值和未来检索意图做出的编辑判断。

核心问题：

> 用户以后想找回这篇笔记时，最自然会去哪个抽屉？

先推断多个维度：

```text
主要用途：学习、参考、决策、行动、记录、理解事件、积累素材、后续研究
内容对象：项目、文章、观点、经验、产品、事件、方法、资料、案例
内容领域：技术、职业、商业、生活、旅行、消费、学习等，由内容决定，不预设
相关场景：学习、工具使用、产品选择、旅行计划、技术实践等
主题关键词：更细的概念、公司、项目、地点、产品、方法
```

然后只选择一个主检索维度作为目录路径。

### 目录规则

```text
目录路径表达主检索路径。
topics 和 tags 表达交叉属性。
优先复用已有目录，但前提是已有目录能覆盖当前内容的主要检索意图；不要为了复用而放入语义错误的目录。
没有自然目录时，创建宽泛目录。
默认使用两级目录。
不主动创建三级目录。
不做目录重构。
不确定时选择更宽、更稳定的目录，而不是创建过细目录。
```

没有自然目录时，读取 `references/classification-index-template.md` 中的「新建目录规则」，按其中的约束和示例创建宽泛目录。

### 宽泛目录原则

新建目录时不要过细。

例如，一篇蓝牙耳机笔记可能同时涉及：

```text
消费
数码
音乐
通勤
运动
配件
```

不要创建：

```text
生活/消费/数码/音乐/蓝牙耳机
```

应选择主要检索路径，例如：

```text
生活/数码设备
```

或者复用更自然的已有目录。其他属性放入 `topics` 和 `tags`。

### 分类边界示例

这些示例用于指导判断，不是固定分类表。

```text
旅行 vs 生活：
如果核心是目的地、路线、交通、住宿、景点、签证、预算、行程规划，倾向旅行。
如果核心是长期居住、城市生活、日常经验、生活成本、居住方式，倾向生活。
如果两者都有，按未来检索意图判断。

消费 vs 生活：
如果核心是购买、选择、参数、使用体验、避坑、对比，倾向消费/装备。
如果核心是日常流程、习惯、居住、健康、关系、生活方式，倾向生活。
数码、家电、户外、运动等不要全部混进一个大杂烩；应根据已有目录和主检索意图选择较宽但可区分的子类。

技术工具 vs 技术学习：
如果核心是一个工具、项目、repo、软件、工作流，倾向工具/工作流。
如果核心是概念解释、原理学习、教程知识，倾向学习/知识。
如果核心是工程实践、架构、踩坑，倾向工程实践。

技术中的不同内容：
工具评测放工具/效率相关目录。
教程放学习/知识相关目录。
项目源码放工程实践相关目录。
行业动态放技术动态或行业观察相关目录。
不要因为都属于技术就混在一个目录里。

公司/财报/商业事件：
如果主要是理解公司业务、财务、行业趋势，倾向商业/公司分析。
如果主要是投资决策素材，且已有投资相关目录，倾向投资/公司研究。
如果只是短期新闻，不为单个公司新建目录，除非该公司是长期关注对象。
```

## tags

用于 Obsidian 原生 Tag Pane 浏览和 `tag:` 搜索。回答两个问题：这属于什么领域？这是什么类型的内容？

规则：

```text
默认使用 3-6 个 tags。
领域标签 + 内容类型标签，不放具体公司名、产品名、项目名（除非该实体是长期关注对象）。
优先复用 classification_index.md 中已有 tags。
tags 应可复用、准确、克制。
```

命名规则：

```text
中文标签可以直接使用中文。
英文词组可以使用短横线，例如 AI-Agent。
中英文混合标签可以使用下划线，或沿用已有写法。
优先复用已有 tag 写法，不创造近义变体。
```

## topics

细粒度概念索引，表达笔记涉及的具体概念、方法、项目、产品等。不进入 Obsidian 原生 tag 系统，用于 Dataview 查询和未来 wiki 编译。

规则：

```text
默认使用 6-15 个 topics。
可以放：项目名、产品名、方法名、技术术语、公司名、地点、具体数字或编号。
不需要刻意复用，每篇独立。
topics 内容不需要和 tags 去重——tags 是粗粒度浏览入口，topics 是细粒度概念索引。
```

## Wiki 字段判定

`wiki_candidate`、`wiki_priority`、`wiki_status` 由 agent 初判，用户后续可手动修改。

- **wiki_candidate: true** — 内容包含可复用概念、方法、项目、工具、案例、数据、长期主题，值得后续沉淀。
- **wiki_candidate: false** — 一次性信息、低价值闲聊、短期资讯、个人临时记录。
- **wiki_priority: high** — 高信息密度、长期复用价值强、与已有知识主题强相关。
- **wiki_priority: medium** — 有一定复用价值，但不紧急。
- **wiki_priority: low** — 可能有用但价值较弱。
- **wiki_status** — 默认 `pending`，用户后续手动推进。

## Frontmatter 禁止字段

不要将以下信息放进笔记 frontmatter：

```text
source_url
source_author
source_time
source_type
source_platform
input_dir
classification_reason
wiki_reason
all_local_assets
source_role_details
```

来源和素材追溯放在 `来源与素材` 小节。机器追踪字段放在 `content_registry.json`。
